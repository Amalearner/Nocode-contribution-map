Step 3 of 4

Recognition Systems

What gets recognized gets repeated. These practices ensure non-code contributors feel their work matters as much as any pull request.

🤖
All-Contributors
A bot and spec that adds every contributor to your README — not just code authors. Recognizes docs, design, talks, translations, and more with emoji-coded contribution types.
📣
Release Shoutouts
Name non-code contributors in every release announcement. When someone fixes a typo that unblocked 200 users, say so publicly in the changelog.
🏷️
Contributor Roles
Create explicit roles like Community Manager, Documentation Lead, or Accessibility Advocate. These titles are visible, meaningful, and give contributors a home.
🎤
Platform Access
Invite non-code contributors to speak at project events, write for the blog, or co-host office hours. Giving voice is recognition in action.
🌱
Mentorship Pathways
Pair newcomers with an existing contributor for their first few months. A mentor's investment is recognition that says "you belong here and we'll invest in you."
📋
Contribution Tracking
Maintain a public record of impact — event attendees reached, docs improved, issues responded to. Data makes invisible work visible.

⚠️ Anti-Patterns to Avoid

These common mistakes actively discourage non-code contributors from staying:

Code-only changelogs Only listing PRs erases the documentation, design, and community work that shipped alongside them.
No contribution guide If someone has to ask "can I help with X?", there's no guide. Silence reads as "non-coders not welcome."
Code of Conduct theater Having a CoC that's never enforced is worse than none — it signals safety, then betrays it.
Gatekeeping by default "If you want to help, open a PR" assumes coding fluency. Inclusive projects define multiple doors.
Step 4 of 4

Workshop Activities

60 minutes. Five activities. One goal: leave with a real plan for making your project more inclusive.

0–5 min
01
Warm-Up: Skills Inventory
Each person lists 3 skills they bring outside of coding. Share with a neighbor. Notice the diversity already in the room.
5–20 min
02
Map Exploration
In pairs, explore the Contribution Map. Mark which categories your project currently supports. Circle the gaps. What's missing?
20–40 min
03
Journey Design Sprint
Groups of 3–4 design a contributor journey for a fictional (or real) project. Define Discover → Orient → Contribute → Belong for one non-code role.
40–55 min
04
Gallery Walk & Critique
Groups post their maps. Everyone walks, adds sticky notes with "⭐ strengths" and "❓ questions." Presenter answers in 90 seconds.
55–60 min
05
One Commitment
Each person writes one thing they'll change in their project this week. Shared aloud or posted to the group channel.
